Knowing how Flotick is structured makes every other page in these docs easier to follow.

## Organization

An **Organization** is the top-level tenant. It owns members, projects, sprints, tasks, attendance, leave, payroll, and billing. Every organization has:

- A unique **slug** (e.g., `acme-co`) used in every dashboard URL: `app.flotick.com/acme-co/...`
- A **plan** — Free, Pro, or Max — that controls limits and advanced features
- **Policies** for workflow, attendance, and leave that apply to every member

Data is strictly isolated per organization. Members of one organization cannot see another's data.

## Members and roles

A **Member** is a user inside an organization. Every member has exactly one role:

| Role | What they can do |
| :--- | :--- |
| **Admin** | Everything: organization settings, members, billing, all projects, payroll. Exactly one Admin per organization. |
| **Manager** | Create and manage projects, sprints, and tasks. Approve attendance and leave. Cannot edit billing or change the Admin. |
| **Member** | Work on assigned tasks, log time, request leave, view their analytics, and collaborate. |

Members can also be grouped into **Departments** and **Sub-departments** for reporting and routing.

## Projects

A **Project** is a container for related work. Projects own sprints, tasks, files, and environment configurations.

Key project attributes:

- **Visibility** — `Public` (visible to everyone in the org) or `Private` (visible only to assigned project members)
- **Status** — `Active`, `On Hold`, `Archived`, `Cancelled`, `Completed`
- **Health** — automatically calculated as `On Track`, `At Risk`, or `Off Track`
- **Project lead** — the accountable owner

## Sprints

A **Sprint** is a time-boxed iteration inside a project, with a goal, capacity in story points, and a lifecycle: `Planning → Active → Review → Retrospective → Completed`.

Only one sprint per project can be active at a time. When you complete a sprint, Flotick calculates the velocity automatically from completed story points.

## Tasks

A **Task** is the atomic unit of work. Tasks have:

- A **type**: `Story`, `Bug`, `Task`, `Epic`, or `Spike`
- A **status** that flows: `Backlog → Sprint Backlog → Todo → In Progress → Review → Testing → Blocked → Done`
- A **priority**: `Critical`, `High`, `Medium`, `Low`
- Up to four people: **assignee**, **reviewer**, **tester**, **reporter**
- **Story points**, **labels**, **dates**, **attachments**, **threaded comments**, **work logs**, and **dependencies**

Long-form fields like description and acceptance criteria are encrypted at rest.

## Workforce records

Two records ride alongside delivery:

- **Attendance** — one record per member per day (`Checked In`, `Checked Out`, `Pending`, or `Leave`)
- **Leave** — typed requests (Privilege, Sick, Casual, Earned, Maternity, Paternity, Comp-off, Loss of Pay) tied to balances and the approval workflow

When leave is approved, Flotick creates the matching attendance rows automatically.

## Real-time and notifications

Every meaningful action — task moved, comment posted, sprint started, leave approved — is broadcast in real time over the web app and surfaced in the **Inbox**. Members can opt in to receive selected notifications via Telegram.

## How everything fits together

```
Organization
├── Members (Admin / Manager / Member)
│   └── Departments → Sub-departments
├── Projects (Public or Private)
│   ├── Sprints (one Active at a time)
│   │   └── Tasks → Comments, Work Logs, Files, Links
│   └── Environments → Variables (encrypted)
├── Attendance (one per member per day)
├── Leave Requests → Leave Balances
├── Payroll Runs → Payslips → Loans
└── Files (S3-backed, presigned uploads)
```
