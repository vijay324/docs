---
title: "Sprints"
description: "Plan time-boxed work, track scope and burndown, and carry unfinished Tasks forward."
---

A Sprint belongs to one Project. It includes a title, goal, dates, capacity, visibility, members, Tasks, comments, files, and delivery metrics.

## Lifecycle

`Planning → Active → Review → Completed`

Status changes advance one stage at a time. Planning Sprints can be edited; starting, advancing, completing, and deleting a Sprint require the appropriate delivery permissions.

## Create and start a Sprint

<Steps>
  <Step title="Create the Sprint">
    Choose a Project, dates, goal, capacity, visibility, and members. The start date cannot be in the past, the end date must follow it, and overlapping Sprints in the same Project are rejected.
  </Step>
  <Step title="Add Sprint Tasks">
    Create Tasks in the Sprint or move eligible Project backlog Tasks into it.
  </Step>
  <Step title="Start">
    Move the Sprint from `Planning` to `Active`. Flotick begins recording scope changes and burndown data.
  </Step>
</Steps>

## Complete a Sprint

Velocity is the total story points on Tasks in `Done`. When incomplete Tasks remain, choose whether to move them to another non-completed Sprint or return them to the Project backlog. Flotick records their carry-over origin.

Completed Sprints support a retrospective with what went well, what could improve, action items, and a rating. Sprint pages also expose burndown, scope changes, velocity history, comments, and member contribution data where available.

## Plan the Sprint

Set a clear goal and capacity before moving a Sprint to `Active`. Capacity is expressed in story points and serves as the baseline for velocity and predictability. Add only work that belongs to the selected Project. Flotick rejects invalid date ranges and overlapping Sprints so the Project does not run conflicting time boxes.

## Monitor delivery

During an active Sprint, the detail page provides:

- **Burndown:** completed and remaining points against the Sprint dates.
- **Scope ledger:** additions, removals, point changes, completion, and reopening events recorded while the Sprint is active.
- **Task list and board:** current status of Sprint work.
- **Velocity and contribution data:** points completed and historical context where the selected view supports it.
- **Comments and files:** a shared record for decisions, demos, and Sprint context.

Scope changes are recorded rather than silently ignored. They are useful for understanding why delivered work differs from the original commitment.

## Carry unfinished work

When completing a Sprint, review incomplete Tasks before confirming. You can send selected or all incomplete work to an eligible next Sprint, or return it to the Project backlog. Carried Tasks retain a reference to the Sprint they came from, which makes carry-over visible in later reviews.

## Retrospectives

After the Sprint is completed, authorized members can create one retrospective. Capture what went well, what needs improvement, action items, and a rating. The retrospective is stored with the Sprint so the team can revisit it during later planning.
