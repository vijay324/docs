title: "Env Storage"
description: "Store encrypted environment configuration by Project, codebase, and workspace."
---

Env Storage is a paid-plan module for Project configuration. Access also depends on your member permission.

## Structure

`Project → Codebase → Workspace → Environment variables`

Codebase types include backend, frontend, mobile, database, DevOps, API, and custom. Workspaces can represent production, staging, development, testing, or a custom environment, with active, inactive, or maintenance status.

## Manage configuration

<Steps>
  <Step title="Select a Project">
    Open **Env Storage** and choose an accessible Project.
  </Step>
  <Step title="Create a codebase and workspace">
    Authorized members can define the codebase, then create one or more workspaces under it.
  </Step>
  <Step title="Add variables">
    Store configuration entries in the selected workspace. Sensitive values are encrypted at rest and masked in the interface.
  </Step>
  <Step title="Use or export">
    Copy values when permitted or download the workspace configuration as an `.env` file.
  </Step>
</Steps>

Deleting a codebase or workspace removes its nested configuration after confirmation. Use separate workspaces to prevent production values from being mixed with development or testing.

## Create a codebase

Select the Project, open the **Codebases** tab, and click **Add Codebase**. Provide a name, choose the closest codebase type, and save. A Project can contain multiple codebases, such as a frontend and API, so each one can keep its own environment structure.

## Create a workspace

Select a codebase, then create a workspace. Give it a name and choose its type and status. Use separate `Production`, `Staging`, `Development`, and `Testing` workspaces when the configuration values or access expectations differ.

## Manage variables

Open a workspace to add, edit, or remove environment configuration entries. Mark sensitive values as secrets. Secret values are encrypted at rest and presented as masked values in the workspace. Copy and export controls apply to the selected workspace, so verify the environment before using them.

## Access and safety

Only accessible Projects appear in Env Storage. Admins, Managers, and eligible Project creators can manage Project environments; members need the specific Env Storage permission to view the module. Store values carefully: exporting an `.env` file puts the selected values on the device that downloads it.

<Warning>
Never paste production secrets into a Task, comment, feature request, or support email. Use the intended Env Storage workspace instead.
</Warning>
